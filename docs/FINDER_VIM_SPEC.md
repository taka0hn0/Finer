# Finer 要件定義・基本設計

- Status: Draft 0.1
- Last updated: 2026-07-18
- Source of truth: This document

## 1. 目的

macOS FinderをVim風に操作できる、高速・軽量・設定可能なキーボード操作環境を提供する。

プロダクトの最上位原則は次のとおりとする。

> 使っている間はネイティブ級に速く、使っていない間は専用リソースを消費しない。

英語での基本メッセージは次を候補とする。

> Fast when active. Zero footprint when idle.

## 2. プロダクト価値

優先順位は以下の順とする。

1. 高速で滑らかな操作
2. アイドル時の軽さ
3. Finderの複数表示形式への対応
4. UI言語に依存しない動作
5. ユーザーが変更可能なキーバインド
6. ファイル操作の安全性
7. 簡単な導入・更新・削除

機能数だけで競合するのではなく、速度・軽さ・堅牢性を製品特性とする。

## 3. スコープ

### 3.1 対象

- Finderの通常ウィンドウ
- Finder Desktop
- List View
- Column View
- Icon View
- 日本語UIと英語UI
- Normal Mode
- Visual Mode
- 非連続マーク
- ファイルのコピー、移動、削除、名前変更
- Vim風モーションとカウント

### 3.2 初期リリースの対象外

- Finder以外のアプリに表示されるOpen/Saveダイアログ
- Gallery Viewの正式対応
- WindowsやLinux
- ネットワーク同期サービス
- 常時表示するメニューバーアプリ
- 利用状況テレメトリ

対象外項目は将来の検討を妨げない。

## 4. 機能要件

### 4.1 ナビゲーション

- `FR-NAV-001`: `h/j/k/l` でFinder内を移動できること。
- `FR-NAV-002`: キー長押しで滑らかに連続移動できること。
- `FR-NAV-003`: List、Column、Iconの表示形式ごとに自然な移動規則を持つこと。
- `FR-NAV-004`: 先頭・末尾・行端で設定に従って折り返せること。
- `FR-NAV-005`: 数値プレフィックスで移動回数を指定できること。
- `FR-NAV-006`: フォルダとファイルが混在しても移動が停止しないこと。
- `FR-NAV-007`: FinderがAX階層を再構築した場合にコンテナを再取得できること。
- `FR-NAV-008`: 大量の未処理入力を残さず、キーを離したら直ちに停止すること。
- `FR-NAV-009`: Normal Modeの`o`で、選択項目をFinder標準の「開く」操作へ渡せること。
- `FR-NAV-010`: List Viewの`h/l`は、Finderが提供する開示三角形の折りたたみ・展開を含む標準の左右移動を維持すること。

### 4.2 モードと選択

- `FR-MODE-001`: Normal ModeとVisual Modeを提供すること。
- `FR-MODE-002`: 非連続項目をマークできること。
- `FR-MODE-003`: Escapeでモード、カウント、マーク、保留操作を安全に解除できること。
- `FR-MODE-004`: テキスト入力欄では通常の文字入力を妨げないこと。
- `FR-MODE-005`: ダイアログ、シート、無効なAX状態では破壊的操作を実行しないこと。
- `FR-MODE-006`: 常駐監視を使わない場合、各コマンドの実行直前に実際のFinder状態を検証すること。
- `FR-MODE-007`: `s`で非連続マークを切り替えた後の最初の`h/j/k/l`は、Finderが返す複数選択の順序ではなく、最後に`s`を押した項目を起点にすること。
- `FR-MODE-008`: Normal Modeの`Esc`はFinderの場所を移動せず、非連続マークとFinderの通常ウィンドウまたはDesktop上の表示選択を解除すること。
- `FR-MODE-009`: Visual Modeは`v`を押した時点の項目を始点として保持し、`h/j/k/l`、数値付き上下移動、`gg`、`G`で始点から移動先までの範囲選択を更新できること。
- `FR-MODE-010`: Normal Modeで確定マークがある間は、確定マークを表示選択として残し、`h/j/k/l`で動く現在位置を一時カーソルとして追加表示すること。移動時は直前の一時カーソルだけを置き換え、確定マークへ昇格させないこと。
- `FR-MODE-011`: Visual Modeの`Esc`は1回でVisual Modeを終了し、Finderの表示選択、Visual始点、非連続マーク、移動カウント、コピー・カット保留状態を解除すること。

### 4.3 ファイル操作

- `FR-FILE-001`: `y`は確定マークがあれば確定マークだけをコピー対象とし、表示上の一時カーソルを含めないこと。確定マークがない場合はFinderの現在選択を対象にすること。
- `FR-FILE-002`: `x`は確定マークがあれば確定マークだけをカット対象とし、表示上の一時カーソルを含めないこと。確定マークがない場合はFinderの現在選択を対象にし、別フォルダへ移動できること。
- `FR-FILE-003`: `d`は確定マークがあれば確定マークだけをゴミ箱へ移動し、表示上の一時カーソルを含めないこと。確定マークがない場合はFinderの現在選択を対象にすること。
- `FR-FILE-004`: 名前変更、新規フォルダ、情報表示、隠しファイル切り替えを提供すること。
- `FR-FILE-005`: 絶対パス、親ディレクトリ、拡張子付き名称、拡張子なし名称をコピーできること。
- `FR-FILE-006`: ファイル操作が成功するまでカット・コピー状態を消去しないこと。
- `FR-FILE-007`: 複数項目の一部が失敗した場合、失敗項目と理由を保持・報告できること。
- `FR-FILE-008`: 同名項目、権限不足、別ボリューム、消失した移動元を安全に扱うこと。

### 4.4 設定

- `FR-CONFIG-001`: ユーザーがキーバインドを設定ファイルで変更できること。
- `FR-CONFIG-002`: 折り返し、長押し閾値、音、HUDを設定できること。
- `FR-CONFIG-003`: 設定からKarabiner用JSONを生成できること。
- `FR-CONFIG-004`: 個人の `karabiner.json` 全体を上書きしないこと。
- `FR-CONFIG-005`: 設定エラーを適用前に検出し、既存の動作可能な設定を保持すること。

### 4.5 診断とヘルプ

- `FR-DIAG-001`: `?` でキーマップと現在のモードを確認できること。
- `FR-DIAG-002`: 診断コマンドでKarabiner、権限、インストールファイル、設定、ソケットを検査できること。
- `FR-DIAG-003`: 通常時はログを増やさず、診断時だけ詳細ログを有効化できること。

## 5. 非機能要件

### 5.1 性能

以下は初期目標値であり、実測後に確定する。

- `NFR-PERF-001`: コールド状態の初回操作を20ms以内で処理することを目標とする。
- `NFR-PERF-002`: バースト中のコマンド処理をp95で8ms以内とすることを目標とする。
- `NFR-PERF-003`: キー入力からFinder選択反映までをp95で16ms以内とすることを目標とする。
- `NFR-PERF-004`: 60秒の長押しで停止、入力残り、意図しない飛びが発生しないこと。
- `NFR-PERF-005`: 1万項目を含むフォルダで基本移動を継続できること。
- `NFR-PERF-006`: Finder標準、主要競合、現行実装と同一条件で比較できるベンチマークを持つこと。
- `NFR-PERF-007`: 通常の1ステップ処理で、項目数に比例する個別AX問い合わせを行わないこと。
- `NFR-PERF-008`: 10、1000、10000項目の測定から、項目数に対するレイテンシの増加率を記録すること。
- `NFR-PERF-009`: AXの一括取得後の検索・照合はプロセス内メモリで行うこと。
- `NFR-PERF-010`: 100ms間隔の独立した`h/j/k/l`入力で、順序の入れ替わり、取りこぼし、入力残りを起こさず、滑らかに選択を反映できること。

測定では平均値だけでなく、p50、p95、p99、最大値を記録する。

### 5.2 リソース消費

- `NFR-RES-001`: アイドル時の専用常駐プロセス数を0とすること。
- `NFR-RES-002`: アイドル時の専用CPU使用率を0%とすること。
- `NFR-RES-003`: アイドル時にポーリングや定期Wakeupを行わないこと。
- `NFR-RES-004`: 操作中ワーカーのPrivate Memoryを10MB未満にすることを目標とする。
- `NFR-RES-005`: 操作終了後、ベンチマークで決めた短いアイドル時間を経てワーカーが終了すること。
- `NFR-RES-006`: 長押し中にステップごとのプロセス生成を行わないこと。
- `NFR-RES-007`: HUDや高度な機能は必要時だけ起動・ロードすること。

「追加RAM 0MB」などの対外表現は、専用プロセスが存在しないアイドル状態を実測したうえで使用する。

### 5.3 信頼性

- `NFR-REL-001`: AX参照が無効化された場合にクラッシュせず再取得すること。
- `NFR-REL-002`: Finder終了、再起動、ウィンドウ切り替えに耐えること。
- `NFR-REL-003`: 状態ファイルをロックし、必要な更新を原子的に行うこと。
- `NFR-REL-004`: インストール、更新、アンインストールを複数回行っても壊れないこと。
- `NFR-REL-005`: 破壊的操作の途中失敗で未処理対象を失わないこと。

### 5.4 互換性

- `NFR-COMPAT-001`: リリース時点の最新2世代のmacOSを対象とすることを目標とする。
- `NFR-COMPAT-002`: 日本語UIと英語UIを正式テスト対象とすること。
- `NFR-COMPAT-003`: メニュー項目名への依存を最小化し、AXRole、Finder API、標準ショートカットを優先すること。
- `NFR-COMPAT-004`: US配列とJIS配列を正式テスト対象とすること。

### 5.5 プライバシーとセキュリティ

- `NFR-SEC-001`: テレメトリや利用状況送信を行わないこと。
- `NFR-SEC-002`: ユーザーのファイル名、パス、クリップボード内容を外部送信しないこと。
- `NFR-SEC-003`: インストーラーが変更するパスを明示すること。
- `NFR-SEC-004`: ダウンロード物にはチェックサムを提供すること。

## 6. アーキテクチャ

### 6.1 基本方針

Karabinerは入力判定に集中させ、Finder固有の状態処理はオンデマンドのネイティブワーカーが担当する。

常時起動するSwift、AppKit、Electronプロセスは採用しない。

```text
Keyboard
  -> Karabiner-Elements
  -> command transport
  -> on-demand C burst worker
  -> Accessibility / Finder
```

### 6.2 Karabinerの責務

- Finderが前面かどうかの判定
- テキスト入力中のバイパス
- 物理キーと論理コマンドの対応
- 必要最小限のmodifierとmode条件
- ワーカーへの短いコマンド送信

### 6.3 ワーカーの責務

- AXコンテナと項目の取得
- 操作バースト中のAXキャッシュ
- 移動、折り返し、カウント
- Finder階層変更時のキャッシュ無効化と再取得
- 選択・マーク・コピー・カット状態の処理
- 実行直前の安全性検証
- エラーコードと診断情報の生成
- AX属性の一括取得とプロセス内での検索・照合
- 項目単位のAX問い合わせを互換フォールバックへ限定

### 6.4 ワーカーのライフサイクル

1. 最初のコマンドで起動する。
2. 操作バースト中は同じワーカーを再利用し、状態変更と移動を受信順に直列実行する。
3. 長押し中は同じAXコンテキストを利用し、無効化時だけ再取得する。
4. Finderが前面でなくなった場合は速やかに終了する。
5. 最後のコマンドから短いアイドル時間が経過したら終了する。
6. 安全上の最大稼働時間を設ける。

ワーカーのアイドル終了時間は750msを公開既定値とする。300ms、500ms、750ms、1000ms、1500msを固定タップ間隔で比較し、プロセス再利用とアイドル時のメモリ保持時間のバランスから決定した。測定時だけ、メトリクスを有効にしたうえで5候補のいずれかへ切り替えられることとする。

確定マークがないListの縦方向長押しは、オンデマンドのバックグラウンド実行プロセスからFinder標準の上下矢印auto-repeatを送る。各反復の直前にキー解放tokenと開始時のFinder PIDを検証し、通常区間ではAX選択readbackやスクロールバー操作を行わない。開始位置から端までの理論ステップ数を越えた後だけ6イベントごとに現在選択を読み、2回連続で位置が変わらなければ端へ到達したと判定する。判定時は矢印key-upを送り、反対側の選択可能な端へAX選択と表示位置を移してから、ネイティブauto-repeatを再開する。キー解放時も必ずkey-upを送り、入力残りを防ぐ。

Columnの縦方向移動は、既定では開始時に現在位置を1回取得し、以後はキャッシュした項目配列と予測位置から移動先だけをAX選択する。各反復の選択readbackは行わず、8.333ms単位の絶対時刻スケジュールで反復する。処理が予定時刻を越えた場合は過ぎたtickを飛ばし、遅れを取り戻すための連続書き込みを行わない。反復中は1要素の可変選択配列を再利用し、キー解放tokenをAX書き込み直前に1回だけ確認する。キーを離した時に実選択を短時間検証する。

`finder_native_column_hold_experiment=1`の時だけ、確定マークがないNormal ModeのカウントなしColumn `j/k`をKarabinerからFinder標準の上下矢印へ直接写像する。ColumnはFocused UI Elementの`role=AXList`かつ`subrole!=AXCollectionList`で判別し、Icon Viewを除外する。Visual Mode、確定マーク付き移動、数値移動、Icon Viewは既存の検証付き経路を維持する。実験経路はFinder標準と同じく列の端で停止し、Column用の端折り返しは物理入力で通常区間を検証した後に別途設計する。

### 6.5 コマンド転送

初期実装は現行のUNIX datagramと一時ワーカー方式を利用できる。

`s`によるマーク更新と直後の`h/j/k/l`は同じ一時ワーカーへ送り、マーク選択とアンカー保存が完了するまで後続移動を開始しない。

一時ワーカーは起動から終了まで専用lockを保持する。クライアントはUNIX datagram socketへの送信成否だけでワーカーの生存を判断せず、lockを取得できた場合は受信プロセスが存在しないsocketを削除して新しいワーカーを起動する。lockを取得できない場合だけ既存socketへの送信を再試行する。

ワーカーは確定マーク一覧と一時カーソルを別状態として扱う。確定マークがある間の移動では、表示選択を「現在の表示範囲にある確定マーク + 移動先カーソル」として更新する。`y/x/d`はファイル上の確定マーク一覧を優先して読み、一覧が空の場合だけFinderの現在選択へフォールバックする。

最適化候補として、Karabinerの `send_user_command` とlaunchd socket activationを検証する。ただし以下を満たす場合だけ採用する。

- 専用常駐プロセスを必要としない。
- コールドスタートが現行方式より悪化しない。
- インストールとアンインストールが安全である。
- ソケットへ送信した入力を取りこぼさない。

### 6.6 表示形式別戦略

ネイティブ矢印イベントとAX直接選択を実測比較し、表示形式別に速く安全な方式を採用する。

- List: 単押し、数値移動、確定マーク付き移動では生の`AXRows`を一括取得し、移動先だけをAX選択する。選択できないグループ見出しなどに当たった場合だけ次候補へ進み、各行の子要素検証は行わない。確定マークなしの縦長押しはFinder標準の上下矢印auto-repeatを使い、通常区間の表示追従をFinderへ任せる。端だけ停止を限定的にAX確認し、反対端の選択と表示位置へ折り返す。Normal Modeの`gg/G`は先頭・末尾を選択したうえで縦スクロール位置も同じ端へ動かし、選択項目を必ず表示する。`h/l`はFinder標準の左右移動として開示三角形の折りたたみ・展開を維持し、選択項目を開く操作は`o`から`Command+Down Arrow`を送る。
- Column: Finder標準移動を優先し、階層更新を検出する。既定の同一列内縦移動は予測位置型AX経路を使う。独立した既定OFF実験では、Focused UI Elementが`AXList`かつ`AXCollectionList`でない時に限り、確定マークなし・カウントなしNormal Modeの`j/k`を標準上下矢印へ直結する。
- Icon: `AXCollectionList`直下の`AXSectionList`を1段展開し、各sectionの前後にあるURLを持たない補助要素だけを除外する。補助要素の判定では全項目を走査せず、section両端から最初のファイル項目までだけを調べる。Finderが公開する項目配列は表示位置に応じて仮想化されるため、`h/l`は同一行では標準矢印を送り、行境界では上下移動と行端への水平移動を組み合わせる。表示範囲が変わった時だけコンテキストを再取得し、フォルダ全体の先頭・末尾では縦スクロール位置を反対端へ動かしてから、その端で公開された最初・最後のファイルを選択する。水平イベント後は算出済み移動先を次コマンドの予測位置として保持する。
- 境界移動: 必要に応じて先頭・末尾を直接選択する。

## 7. 禁止事項

- 専用プロセスを `KeepAlive` で常駐させない。
- アイドル時のポーリングを行わない。
- キーリピートの各ステップでシェルやヘルパーを起動しない。
- 個人のホームディレクトリを絶対パスで埋め込まない。
- 個人の `karabiner.json` 全体を配布しない。
- 成功確認前にカット・コピー状態を破棄しない。
- 性能・メモリの未計測値を宣伝文句に使わない。
- 大規模な書き直しをベンチマークなしで採用しない。

### 8.1 比較対象

- Finder標準キー操作
- chrisgrieser/finder-vim-mode
- 現行の `finder_ax_step`
- 新しいワーカー実装

### 8.2 テストケース

- 10、1000、10000項目
- フォルダのみ、ファイルのみ、混在
- List、Column、Icon
- 単押し、連打、長押し、折り返し
- フォルダからファイルへの境界移動
- Finder階層の更新

### 8.3 測定項目

- コールドスタート時間
- ウォーム状態のコマンド処理時間
- キー入力から選択変更までの時間
- キーを離してから停止するまでの時間
- CPU時間
- RSSとPrivate Memory
- Wakeup回数
- プロセス生成回数
- 入力取りこぼしと入力残り
- 1コマンド当たりのAX問い合わせ回数
- 項目数に対するレイテンシ増加率

### 2026-07-17: 一時ワーカーの生存判定はsocketではなくworker lockを正とする

- Decision: コマンドクライアントは一時ワーカーが生存している間だけ保持される専用lockを先に確認する。lockを取得できない時だけ既存UNIX datagram socketへ送信し、lockを取得できた時は残存socketを削除して新しいワーカーを起動する。

### 2026-07-17: Listのマークなし縦長押しはネイティブ矢印を使い、端だけAXで折り返す

- Decision: Normal ModeのList Viewで確定マークがない`j/k`長押しは、LaunchServicesからオンデマンド起動するバックグラウンド実行プロセスがFinder標準の上下矢印auto-repeatを送る。通常区間はAX選択readbackを行わず、Finder自身の選択描画とスクロール追従を使う。端までの理論イベント数を越えた後だけ6イベントごとに選択位置を読み、2回連続で変化しなければ矢印key-up、反対端へのAX選択、縦スクロール位置の端移動を行ってauto-repeatを再開する。各イベント前と折り返し直前にキー解放tokenと前面Finder PIDを検証する。
- Reason: 予測位置へ直接AX選択する経路は移動速度を改善したが、List Viewで選択が表示範囲外へ進み、Finder標準の矢印長押しと同じ滑らかなスクロール追従にならなかった。全区間をネイティブ矢印へ任せると表示追従は得られる一方、Finder標準は端で停止するため、Finerの折り返し要件を端だけのAX介入で補う。

### 2026-07-17: Columnのマークなし縦長押しと旧List候補は予測位置型AX経路で反復する

- Decision: 確定マークがないColumnの`j/k`長押しは、長押し開始時に現在位置を1回だけ読み、キャッシュ済み項目配列内の予測位置を更新しながら移動先へ直接AX選択を書き込む。反復は8.333ms単位の絶対時刻で予定し、AX処理がtickを越えた場合は過ぎたtickを飛ばして次の未来tickを待つ。hot loopでは1要素の可変選択配列を再利用し、キー解放tokenを各AX書き込みの直前に1回だけ読み、開始時に記録したFinder PIDと前面PIDを毎回比較する。キーを離した時だけ実選択を検証し、選択が空なら予測位置から同じ方向へ選択可能項目を探索して確定する。確定マークがある場合はマーク表示と一時カーソルを守る既存の検証付き経路へ戻す。Listに対する同方式は性能比較の旧候補として記録を残すが、通常の表示追従を満たすため直前のネイティブ矢印Decisionで置き換える。

- Constraint: この測定は物理キー、Karabiner評価、Finderの各フレーム描画を含まず、groupingも制御していないため、ネイティブ同等のend-to-end性能を示すものではない。初回のthroughput比較は相対待機版であり、絶対時刻化とhot-loop割り当て削減後の速度・滑らかさはdogfoodと画面収録で別途確認する。前面確認の頻度とmovement lockの範囲は縮小せず、単押し、100ms連打、数値移動、Icon View、確定マーク付き移動の意味は変更しない。grouped Listでは長押し終了時の選択が空でないことを専用回帰で検証する。

### 2026-07-17: 確定マークを表示に残し、ファイル操作から一時カーソルを除外する

- Decision: Normal Modeで確定マークが存在する間、Finderの表示選択を「現在の表示範囲にある確定マーク + 現在位置の一時カーソル」とする。`h/j/k/l`、数値移動、`gg/G`は最後に`s`を押した項目または保存済み一時カーソルを起点にし、移動後も確定マークを青い選択として残す。`y/x/d`は確定マークが1件以上あれば確定マークだけを対象にし、一時カーソルを含めない。確定マークがない時だけFinderの現在選択を対象にする。Visual Modeの`Esc`は1回でモードと表示選択をすべて解除する。
- Reason: Finderの選択表示だけでは、ユーザーが`s`で確定した項目と移動中の現在位置を区別できない。確定マークを表示に残せば次に操作される集合を確認できる一方、青く見える一時カーソルまで`y/x/d`へ含めると、確定していない項目を誤操作する。Visual Modeだけ2段階の`Esc`にする用途もなく、1回で安全に選択を破棄する方が予測しやすい。
- Constraint: Finderは確定マークと一時カーソルを同じ青色で表示するため、区別はFinerのマークファイルとナビゲーションアンカーで保持する。表示選択配列の順序を操作対象の決定に使わない。状態更新と移動は既存のオンデマンドワーカー内で行い、常駐処理を追加しない。マークファイルを読まないKarabiner直結の高速実験は、`s`が1回でも押された後から`Esc`またはVisual Mode開始でマークを消去するまで保守的に無効化し、マーク対応ワーカーへ戻す。List・Column・Iconの専用Finderウィンドウ試験で、AとBを確定後にCへ移動した表示がA+B+C、コピー対象がA+Bのみ、`Esc`後の表示選択が空になることを検証する。

### 2026-07-17: Karabiner配布JSONは機能別source moduleから決定的に生成する

- Decision: `Finer Utility Commands`と`Finer Navigation`を独立した`rules/source/*.json`のrule objectとして正本にし、固定順で`Finer (development snapshot)`へ結合した`rules/generated/finder-vim.json`をimportable snapshotとして追跡する。通常の`make check`は再生成結果とのbyte-for-byte一致と隔離した生成・stale検出・symlink拒否を検証する。
- Reason: 5000行を超える配布JSONだけを直接編集する状態から、機能単位の正本と生成物を分離し、source変更後の生成忘れやdogfood環境からの逆流をCIで検出するため。既存JSONを2つのrule objectへ機械的に分割して再結合した結果、生成前後のSHA-256 `87455f79b8f035ac1b77112fb892312f2db66ed426648e82de35d67ff27407a6`が一致することを初期移行条件とする。
- Constraint: この段階は現行キーマップを変更せず、ユーザー向け設定ファイル形式や任意キーバインドのschemaを確定しない。`FR-CONFIG-001`から`003`のユーザー設定生成は別変更で設計し、install先は引き続き検証済みのgenerated snapshotだけを受け取る。

### 2026-07-17: 初期配布物はコミット由来の再現可能なソースアーカイブとする

- Decision: pre-alphaの配布物は、指定したGitコミットを`git archive`で単一の`Finer-<version>/`ルートへ書き出し、mtimeを追加しない`gzip -n`で圧縮したソースアーカイブと、隣接するSHA-256ファイルとする。同じコミットとversionから2回生成したアーカイブがbyte-for-byteで一致すること、パスが単一ルート外へ出ないこと、展開後の`make check`と隔離install/uninstallが成功することを自動検証する。
- Reason: 初期段階の「ソースからローカルビルド」という配布要件を満たしながら、作業ツリーの未コミット内容、個人ファイル、`.git`、圧縮時刻による差を配布物へ混入させず、公開チェックサムを再生成可能にするため。
- Constraint: この決定は署名済みバイナリ、notarization、Homebrew Formulaの方式を確定しない。通常の`dist`はtracked変更がある状態を拒否し、明示的な開発検証時だけ`FINER_DIST_ALLOW_DIRTY=1`で未変更の指定refをアーカイブできる。更新は引き続き明示的なユーザー操作とし、常駐アップデーターを追加しない。

### 2026-07-17: 画面反映時間は可変フレームPTSと同画面マーカーで測る

- Decision: Finderの画面に選択変化が現れるまでの自動計測は、固定範囲の画面収録へ非アクティブな赤/緑マーカーを重ね、緑へ更新した直後に既存ヘルパーを起動する。最後の赤フレームを基準に、最初の緑フレームとマーカー外で選択変化を検出した最初のフレームのPTS差を記録する。内部worker metricsも同じコマンドで併記するが、画面値と内部値を同一視しない。
- Evidence: このMacの`/usr/sbin/screencapture`でFinderウィンドウを1秒収録した初期確認はH.264、nominal 60fps、0.883秒に48フレームとなり、静止区間の重複フレームが省かれる可変フレームレートだった。このため、フレーム番号の60fps換算や録画プロセス起動時刻では入力境界を決められない。200msの既知差を持つ60fps合成動画では、PTS解析が赤0.5秒、応答0.7秒、差200msを60/60フレームで復元した。
- Constraint: マーカー更新はヘルパーの`posix_spawn`より先に行うため、値には最大およそ1収録フレームのマーカー先行と合成量子化が含まれる。物理キー、Karabiner評価、shell command開始までの時間は含まず、完全end-to-end key latencyとは表現しない。赤、緑、応答変化、最終選択パス、worker metricsのすべてが揃わない反復は失敗とする。録画は専用Finderウィンドウの固定範囲だけに限定し、通常利用へマーカー、収録、常駐処理を追加しない。

### 2026-07-17: Column遷移のper-transition AXObserverは採用しない

- Decision: Columnの水平移動後は現行の短時間pollingとreadiness確認を維持する。Finderアプリ全体へ`AXFocusedUIElementChanged`と`AXCreated`を登録して待つper-transition AXObserver候補は本番経路へ残さない。
- Evidence: 同一の一時ビルドでrealistic-mixedの1000項目と10000項目を各10反復比較し、pollingとObserverはいずれも20/20成功、Observerのfallbackは0だった。Observerはwarm `l`の平均AX readを46.4/35.6から9.0、平均Wakeupを9.1/6.4から0へ減らしたが、worker p95は190.435/206.139msから192.365/212.130ms、dispatch p95は310.838/348.668msから314.053/358.640msへ増えた。最大active footprintも1000項目で163,840 bytes、10000項目で98,304 bytes増えたため、速度と単純性を優先して候補を戻した。匿名化したraw dataを`benchmarks/results/2026-07-17-column-phases/`へ追加する。
- Constraint: この判断はコマンドごとにObserverを作成・登録する候補に対するものとする。異なるObserver lifetimeやFinder側仕様変更を将来検討する場合も、最終選択の正しさだけでなくworker/dispatch latency、active footprint、CPU、Wakeupを同一条件で再比較し、速度改善がなければ採用しない。

### 2026-07-17: Column階層移動はFinderのAX公開待ちを維持し、フェーズ計測を明示時だけ追加する

- Decision: Columnの`h/l`後は既存どおり、Finderの新しい列で項目配列と選択項目を解決できるまで後続コマンドを同期する。通常のメトリクスに加えて`FINDER_VIM_COLUMN_PHASE_METRICS=1`を指定した時だけ、コンテキスト検証・再作成、イベント送信、列遷移、項目配列、選択確定を別々に記録する。通常利用時はフェーズ時刻を取得しない。
- Evidence: realistic-mixedの1000項目と10000項目で各10反復の`j l j`を測定し、現行待機と一時的なfocused-container-only待機はいずれも20/20成功した。現行のwarm `l` transition p95は162.731msと191.142msだった。古い列のitem-count照会を省くとその照会は0msになったが、focused-container照会p95が148.525msと147.432msへ移り、worker p95も1000項目で186.524msから190.916ms、10000項目で230.935msから221.168msとなって一貫した改善にならなかった。この候補は戻し、匿名化したraw dataを`benchmarks/results/2026-07-17-column-phases/`へ保存する。
- Constraint: この結果はFinderが列をAXへ公開する境界で最初のpost-event AX問い合わせがブロックすることと整合するが、Finder内部の実装を直接観測したものではない。固定delayや待機削除で内部数値だけを短縮せず、`jlj`の最終選択を優先する。物理キー、Karabiner評価、Finderのフレーム描画は測定対象外である。

### 2026-07-17: 主要表示形式は現行経路を維持し、Column階層移動を次の性能調査対象にする

- Decision: ListとIconは現行の表示形式別common pathを維持する。Columnの階層移動は最終選択の正しさを維持したまま、Finderの列生成待ちと外れ値を分離する次の性能調査対象とする。ファイル容量だけを性能代表値にせず、empty-filesとrealistic-mixedを分けて継続測定する。
- Evidence: macOS 26.5.2 / Finder 26.4で、2 profile、List・Column・Icon、10・1000・10000項目の表示マトリクス180反復、長押し20反復、100msタップ60反復の計260反復がすべて成功した。Listのwarm worker p95は全条件で17.275ms以下、Iconは0.480ms以下だった。realistic-mixed Columnは最終選択に失敗しなかったが、1000項目のdispatch p95が最大713.074ms、10000項目の`l` worker p95が482.061msとなった。匿名化したraw dataを`benchmarks/results/2026-07-17-finder-matrix/`へ保存する。
- Constraint: このrunはFinderの実ウィンドウと最終選択を使うが、物理キー、Karabiner評価、Finderの各フレーム描画時間を含まない。Listのgroupingも制御していないため、ネイティブ同等のend-to-end latencyやgrouped List View対応の根拠には使用しない。

### 2026-07-17: Visual Modeの数値移動と`gg/G`は固定始点からAX範囲選択する

- Decision: `v`を押した時点の項目パスと位置ヒントを専用の一時アンカーへ原子的に保存する。単純な`h/j/k/l`はFinder標準の`Shift+Arrow`を維持する。数値付き`j/k`はNormal Modeと同じ1〜99のカウント入力を受け取り、現在の連続選択と固定始点から現在端を判定して、移動先までを1回のAX範囲選択で更新する。Visual Modeの`gg/G`も固定始点からList・Columnの先頭/末尾、Iconの現在列上端/下端までを範囲選択する。
- Reason: Visual Mode中は従来カウントと`g`プレフィックスのmanipulatorを無効化していたため、`5j`の数字や`gg/G`の文字がFinderへ素通りし、頭文字選択など別の標準動作を起こしていた。大量の`Shift+Arrow`イベントを生成せず移動先を直接選択すれば、入力残りを避けながら既存の数値移動と端移動の意味をVisual Modeでも共有できる。
- Constraint: Visual Modeを`Esc/y/x/d`で終了する時とNormal Modeの`Esc`では専用アンカーを消去する。アンカーは実行時状態として保持し、専用常駐プロセスやポーリングを追加しない。単純な`h/j/k/l`と数値・端移動を混在させても、選択範囲に含まれる固定始点から現在端を復元する。

### 2026-07-16: Normal Modeの`Esc`はFinderのDeselect AllメニューをAX実行する

- Decision: Normal Modeの`Esc`はオンデマンドCヘルパーからFinderのメニューバーをAXで探索し、ショートカット属性が`Option+Command+A`であるメニュー項目へ`AXPress`を実行する。メニュー項目を取得できない場合に限り、システム全体のフォーカスを起点とするAX選択解除へフォールバックする。Karabinerからは選択解除ヘルパーと非連続マーク初期化を1本の`shell_command`内で順番に実行し、その後、移動カウントとカット保留状態を消去する。
- Reason: FinderのDeselect Allメニューを直接実行すると、通常の単一選択とFinerの非連続複数選択を同じFinder標準処理で解除できる。`CGEventPostToPid`による`Option+Command+A`は単純な選択では動作する場合があったが、実利用の4項目選択と検証用の18項目選択ではFinderのメニューショートカットとして処理されなかった。メニューへのAXPressなら他アプリへキーイベントを発生させず、Raycastなどのグローバルショートカットとも競合しない。
- Constraint: Finderが最前面であることを実行直前に確認する。UI言語へ依存しないよう、メニュー名ではなく`AXMenuItemCmdChar`または`AXMenuItemCmdVirtualKey`と`AXMenuItemCmdModifiers`でDeselect Allを特定する。同じ`to`配列へ複数の`shell_command`を分離して置かず、選択解除と状態初期化の実行順を単一コマンドで保証する。Visual Modeの`Esc`を終了だけに限定する当時の挙動は、2026-07-17の「確定マークを表示に残し、ファイル操作から一時カーソルを除外する」決定で置き換える。

### 2026-07-16: 非連続マーク後の移動は最後に`s`を押した項目を起点にする

- Decision: `s`は最後に切り替えた項目のURLと位置ヒントを一時アンカーへ原子的に保存する。`s`と通常の`h/j/k/l`は同じオンデマンドCワーカーへ送り、ワーカーはマーク更新用Swift helperの完了を待ってから後続コマンドを処理する。次のNormal Modeの`h/j/k/l`は通常移動・数値移動・長押しのいずれでもアンカーを一度だけ取得し、その項目へFinderの選択を集約してから移動する。Finerの非連続マーク一覧は移動後も維持し、`Esc`またはマークを消費・破棄する操作ではアンカーも消去する。
- Reason: Finderの`AXSelectedRows`と`AXSelectedChildren`は最後に選択した項目を表す順序を保証せず、Finerもマーク再選択時に表示順で選択配列を再構築するため、従来は複数選択の先頭から移動していた。さらに、`s`だけを別のSwiftプロセスとして起動すると、直後の高速な移動クライアントがマーク選択とアンカー保存を追い越せる。FinderのC側AXが返すファイル参照URL（`file:///.file/id=...`）とSwift側のパスURLは同じ項目でも文字列が異なるため、照合時は両方を実ファイルパスへ解決する。これと同じバーストキューでの順序固定により、Finderの選択順、URL表現、プロセス起動速度のいずれにも依存しない。
- Constraint: 常駐プロセスやポーリングを追加しない。次の移動前に選択をアンカーへ集約する当時の表示挙動は、2026-07-17の「確定マークを表示に残し、ファイル操作から一時カーソルを除外する」決定で置き換える。

### 2026-07-16: 長押し性能はFinderの実選択パスから算出する

- Decision: 長押しベンチマークの移動量と成功判定は、helperが返す内部AX行番号ではなく、既知のfixture順序に対するFinderの実際の最終選択パスから算出する。helperの行番号と実ファイル位置の差は診断値として残す。helper終了直後と25ms、50ms、100ms後の選択を記録し、終了後に変化した反復は入力残りとして失敗にする。
- Reason: 生の`AXRows`を使う新版では、ファイルではない行が含まれて内部AX位置がfixtureのファイル位置より常に1大きかった。Finder選択は終了直後から100ms後まで変化していないにもかかわらず、両位置が同一という旧runnerの仮定によって長押しを誤って失敗判定していたため。
- Evidence: 修正後の各10反復で、新版はempty-filesが10/10成功・p50 74.098 steps/s、realistic-mixedが10/10成功・p50 59.791 steps/sで、終了後driftは20反復すべて0だった。旧版はempty-filesが10/10成功・p50 10.459 steps/s、realistic-mixedが1/10成功・p50 0だった。100msタップでは新版が両profileのList、Icon、Columnを全60反復成功し、旧版はIconの20反復すべてで最終位置を取りこぼした。匿名化した生データを`benchmarks/results/2026-07-16-navigation-ab/`へ保存する。
- Constraint: この測定は物理キー、Karabiner評価、Finderの各フレーム描画時間を含まず、List Viewのgroupingも制御していない。対外的なネイティブ同等性能の根拠には単独で使用しない。

### 2026-07-16: 長押しと100ms間隔のタップ連打を別々に測る

- Decision: 長押しの`hold-repeat`とは別に、100ms間隔で独立した`hold-start`クライアントを起動するタップ連打ベンチマークを持つ。List Viewの`j/k`、Icon Viewの`h/l`、Column Viewの`j/l/j/h/k`階層サイクルを固定シーケンスで測り、投入時刻、クライアントenqueue時間、投入遅れ、最終選択と失敗を記録する。
- Reason: 長押しは単一プロセス内の反復経路だが、キー連打はクライアント起動、UNIX datagram転送、750msワーカー再利用、Finderの非同期表示更新を毎回通るため、ボトルネックと失敗条件が異なる。
- Constraint: この自動計測は物理キー、Karabiner評価、Finderの各フレーム描画時間を含まない。100ms間隔での見た目の滑らかさは、同じ実行中に実画面でも確認する。
- Verification: Finderを開かないpreflightを旧版・新版ヘルパーで通した後、empty-filesとrealistic-mixedの同一条件でA/B実行し、全シーケンスの最終選択、失敗0、投入遅れと体感を比較する。

### 2026-07-16: List Viewの展開と選択項目を開く操作を分離する

- Decision: `h/l`はFinder標準の左右移動として開示三角形の折りたたみ・展開を維持する。Normal Modeの`o`は、選択項目を開くFinder標準ショートカット`Command+Down Arrow`をキーリピートなしで送る。親ディレクトリへ移動する既存の`-`は変更しない。Finderのフォルダ単位の表示設定や開示三角形の有無はFinerから変更しない。
- Reason: FinderはList Viewでも場所や表示状態によって開示三角形を表示しないことがあり、展開操作だけではフォルダへ入れないため。ネイティブの展開挙動を保ったまま、表示状態に依存しない明示的な「開く」操作を用意する。
- Verification: 生成ルールに`o`から`Command+Down Arrow`への単一マッピングがあることを自動検証し、dogfoodでフォルダとファイルを開けること、およびList Viewの`h/l`による既存の折りたたみ・展開が変わらないことを確認する。

### 2026-07-16: install/uninstallを事前検証・原子的置換・隔離テストの対象にする

- Decision: インストーラーは全ソース成果物を事前検証し、各配置先と同じディレクトリへ一時ファイルを作成してから原子的に置換する。既存またはローカル変更済みのimportable ruleは置換・削除前に状態ディレクトリへバックアップする。メインの `karabiner.json` と既存の状態ファイルは変更しない。
- Reason: 欠けたビルド成果物や途中のステージ失敗で動作中のインストールを壊さず、ユーザー変更を失わず、install/uninstallを何度でも安全に実行できるようにするため。
- Verification: 一時 `HOME` で、既存ルールのバックアップ、空白を含むパス、再install、欠損成果物による事前失敗、変更済みルールのuninstall前バックアップ、再uninstall、状態とメイン設定の保持を自動検証する。

### 2026-07-16: ワーカーの既定アイドル終了時間を750msとする

- Decision: オンデマンドCワーカーは最後のコマンドから750ms入力がなければ終了する。ベンチマーク時だけ300ms、500ms、750ms、1000ms、1500msへ切り替え可能とし、通常利用時の設定項目にはしない。
- Reason: 5候補、5タップ間隔、各10反復の250ペアで比較した。750msは100ms、400ms、700ms間隔を全て同一プロセスで処理し、50ペア当たりのプロセス生成数は70だった。1000msも同じ70プロセスでCPUとWakeupに利点がない一方、最終コマンド後の合計保持時間は750msの52.600秒に対して70.102秒だった。500msは保持時間を短縮するが700ms間隔を再利用できず80プロセスを生成し、1500msは60プロセスまで減らす代わりに合計90.086秒保持した。
- Evidence: 全250ペアで失敗とメトリクスdropは0、全候補の計測Wakeupは0、最大physical footprintは2,605,488 bytes以下だった。匿名化した生データと環境情報を `benchmarks/results/2026-07-16-worker-idle-timeout/` に保存する。

### 2026-07-15: 正式なプロダクト名をFinerとする

- Decision: 正式なプロダクト名と公開リポジトリ名を `Finer` とする。READMEなどの対外表記では旧称の「Finder Vim」を使用しない。
- Reason: Vim風キーバインドだけを前面に出さず、高速・軽量・アイドル時に専用リソースを消費しないFinder用ツールという製品価値を伝えるため。
- Compatibility: 既存のパス、環境変数、ファイル名に含まれる `finder-vim` / `FINDER_VIM_*` は技術識別子として当面維持し、変更する場合は互換性を保った移行を別途設計する。

### 2026-07-14: ソースとインストール先を分離する

- Reason: 連続操作中のキャッシュと高速性を維持しながら、アイドル時の専用リソースをなくすため。

### 2026-07-14: 大規模な書き直しは測定後に判断する

- Decision: 現行C実装をベースラインとし、他方式は同一条件で比較してから採用する。
- Reason: アーキテクチャ上の印象ではなく、実測で速度と軽さを判断するため。

### 2026-07-14: 項目数に比例するAX問い合わせを通常経路から外す

- Decision: フォーカスから最寄りの対応コンテナを採用し、選択行・選択子要素を一括取得して、インデックス照合はプロセス内で行う。全項目への個別AX問い合わせは互換フォールバックに限定する。
- Reason: AXプロセス間通信はローカル配列検索より大幅に高コストであり、大量項目フォルダの単押し遅延を項目数へ比例させないため。
- Status: まずdogfoodで回帰と体感を確認し、10、1000、10000項目の計測値で確定する。

### 2026-07-14: 水平移動後はFinderの表示更新と後続入力を同期する

- Decision: ディレクトリ遷移になり得る`h/l`の後は、コンテナまたは項目数の変化を短時間待ち、新しいコンテナへ移った場合は項目配列を取得でき、その配列内で選択項目を解決できることも確認してからAXコンテキストを破棄する。後続コマンドは再取得後のコンテキストだけを操作する。
- Reason: `CGEventPost`は非同期であり、高速な`jlj`などでFinderの表示更新前に次のコマンドが古いディレクトリへ適用される競合を防ぐため。
- Constraint: Icon View内の横移動など、ディレクトリ遷移ではない経路では既存キャッシュを維持する。

### 2026-07-15: ワーカー起動中の入力を同じバーストへ送る

- Decision: ワーカーロック取得後からUNIX datagramソケットのbind完了までに届いたコマンドは、送信側でソケットを短時間再試行する。送信側はワーカーのアイドル終了までロック待ちしない。
- Reason: 高速な`jlj`などが起動直後の競合で複数ワーカーへ分断・並べ替えされ、古いFinderコンテキストへ適用されることを防ぐため。

### 2026-07-15: ベンチマーク計測は明示時だけ有効にする

- Decision: 環境変数で明示したベンチマーク時だけ、コマンド送信時刻、処理時間、CPU、Wakeup、メモリ、AX呼び出し、CGEventを記録する。記録はワーカー終了時までメモリに保持する。
- Reason: 同一バーストの後続入力をファイルI/Oで遅延させず、通常利用時のログ生成と計測カウンターをなくすため。

### 2026-07-15: List Viewは生のAXRowsと対象行だけのAX選択を使う

- Decision: List Viewの`j/k`は生の`AXRows`を一括取得し、移動先の行だけをAX選択する。Finderが非選択行を拒否した場合だけ同じ方向の次候補を試す。全行の子要素検証や全行への選択属性書き込みは行わない。
- Reason: 1000項目でcold操作あたり2007回、10000項目では3万回超のAX readが実測され、Finderの遅延行構築中には検証済み行と`AXSelectedRows`が不一致になったため。ネイティブ矢印も比較したが、選択反映の非同期遅延により高速な連続入力が途中行を境界と誤認した。対象行のAX選択は10・1000項目の既存経路でより高速かつ確定的だった。
- Required regression: 通常・grouped List View、ファイル・フォルダ混在、境界wrap、長押しを検証する。

### 2026-07-15: Icon Viewの水平バーストは予測位置を保持する

- Decision: `subrole=AXCollectionList`をIcon Viewとして通常のColumn View `AXList`と区別し、直下の`AXSectionList`を1段展開してアイコン項目を得る。`h/l`で送った矢印イベントの移動先を、同一バースト内の次コマンドが使う予測位置として保持する。Finderの選択がキャッシュ外へ移った場合はコンテキストを再取得する。
- Reason: 10・1000・10000項目で、Finderは矢印イベントを正しく順次処理する一方、ワーカー中のAX選択は移動前の位置を返し続けた。AXを連続pollすると各コマンド128 read、worker p95で100ms超を消費しながら、3コマンドすべてを位置1と誤記録したため。
- Required regression: 同一行の連続移動、行境界のforward/reverse wrap、表示範囲を越える長押しを検証する。
